## 一、RAG（检索增强生成）

### 1. RAG 架构
```
用户问题
  ↓
Query 处理（改写/扩展）
  ↓
Embedding 向量化
  ↓
向量数据库检索 Top-K 相关文档
  ↓
重排序（Reranker）
  ↓
Prompt 组装: 系统 Prompt + 检索到的上下文 + 用户问题
  ↓
LLM 生成回答
  ↓
后处理（引用标注/事实核查）
```

### 2. 文档处理流程
```typescript
// 1. 文档加载
import { PDFLoader } from 'langchain/document_loaders/fs/pdf';
import { TextLoader } from 'langchain/document_loaders/fs/text';
const docs = await new PDFLoader('report.pdf').load();

// 2. 文本分割
import { RecursiveCharacterTextSplitter } from 'langchain/text_splitter';
const splitter = new RecursiveCharacterTextSplitter({
  chunkSize: 500,      // 每块大小
  chunkOverlap: 50,    // 重叠部分（保持上下文连贯）
  separators: ['\n\n', '\n', '。', '！', '？', ' ', ''],
});
const chunks = await splitter.splitDocuments(docs);

// 3. 向量化
import { OpenAIEmbeddings } from '@langchain/openai';
const embeddings = new OpenAIEmbeddings({ model: 'text-embedding-3-small' });

// 4. 存入向量数据库
import { PineconeStore } from '@langchain/pinecone';
await PineconeStore.fromDocuments(chunks, embeddings, { pineconeIndex });
```

### 4. RAG 优化策略
```
检索质量:
  ├── 查询改写（Query Rewriting）
  ├── HyDE（假设文档嵌入）
  ├── 混合检索（向量 + BM25）
  ├── 重排序（Reranker）
  └── 元数据过滤（按来源/时间/类别过滤）

生成质量:
  ├── Prompt 工程（引用来源、结构化输出）
  ├── 上下文窗口管理（按相关性排序）
  ├── 幻觉检测（对比原文验证）
  └── 多轮对话记忆

工程优化:
  ├── 文档增量更新
  ├── 缓存热门查询
  ├── 异步 Embedding
  └── 评估体系（命中率、忠实度、答案相关性）
```

## 二、Agent（智能体）

### 1. Agent 核心概念
```
Agent = LLM + 工具 + 记忆 + 规划

LLM: 大脑（推理和决策）
工具: 手（执行动作）— 搜索、API、代码执行、文件操作
记忆: 上下文（短期: 对话历史，长期: 向量存储）
规划: 思维链（ReAct、Plan-and-Execute）
```

```typescript
// 余弦相似度（最常用）
function cosineSimilarity(a: number[], b: number[]): number {
  const dot = a.reduce((sum, ai, i) => sum + ai * b[i], 0);
  const normA = Math.sqrt(a.reduce((sum, ai) => sum + ai * ai, 0));
  const normB = Math.sqrt(b.reduce((sum, bi) => sum + bi * bi, 0));
  return dot / (normA * normB);
}

// 其他: 欧氏距离、点积、曼哈顿距离
```

### 2. 流式响应实现
```typescript
// API Route (Next.js App Router)
export async function POST(req: Request) {
  const { messages } = await req.json();

  const stream = await openai.chat.completions.create({
    model: 'gpt-4o',
    messages,
    stream: true,
  });

  // 将 OpenAI stream 转为 ReadableStream
  const encoder = new TextEncoder();
  const readableStream = new ReadableStream({
    async start(controller) {
      for await (const chunk of stream) {
        const content = chunk.choices[0]?.delta?.content || '';
        if (content) {
          controller.enqueue(encoder.encode(`data: ${JSON.stringify({ content })}\n\n`));
        }
      }
      controller.enqueue(encoder.encode('data: [DONE]\n\n'));
      controller.close();
    },
  });

  return new Response(readableStream, {
    headers: {
      'Content-Type': 'text/event-stream',
      'Cache-Control': 'no-cache',
      'Connection': 'keep-alive',
    },
  });
}

// 前端消费
const response = await fetch('/api/chat', {
  method: 'POST',
  body: JSON.stringify({ messages }),
});
const reader = response.body.getReader();
const decoder = new TextDecoder();

while (true) {
  const { done, value } = await reader.read();
  if (done) break;
  const text = decoder.decode(value);
  // 解析 SSE 数据并更新 UI
}
```

```typescript
  let totalTokens = countTokens(messages);
  while (totalTokens > maxTokens && messages.length > 1) {
```

## 六、AI 评估与测试

### 1. RAG 评估指标
```
检索质量:
  - Hit Rate: 查询的正确文档是否在 Top-K 中
  - MRR (Mean Reciprocal Rank): 正确文档的平均排名倒数
  - NDCG: 考虑排序位置的指标

生成质量:
  - Faithfulness (忠实度): 回答是否基于检索到的文档
  - Answer Relevance (答案相关性): 回答是否切题
  - Context Relevance (上下文相关性): 检索的文档是否相关

工具:
  - RAGAS — 自动化 RAG 评估框架
  - TruLens — LLM 应用质量评估
  - LangSmith — LangChain 可观测性平台
```

## 七、AI 工程化面试高频题

### Q1: RAG vs 微调，什么时候用哪个？
```
RAG 适合:
  - 知识频繁更新
  - 需要引用来源
  - 专属知识库（公司文档、产品知识）
  - 数据量大但每次只需部分

微调适合:
  - 特定风格/格式要求
  - 特定领域术语
  - 减少推理成本（小模型微调）
  - 稳定的行为模式

可以结合: 先微调基础能力，再用 RAG 注入实时知识
```

### Q2: 如何减少 LLM API 成本？
- 使用更小的模型（GPT-4o-mini、Claude Haiku）
- 缓存常见查询结果
- 压缩 Prompt（减少 System Prompt 长度）
- 只在必要时调用 LLM（规则引擎 + LLM 分流）
- 批量请求
- 本地部署开源模型（高用量场景）

### Q3: Agent 和普通 LLM 调用的区别？
普通调用: 一次性输入输出，无工具使用
Agent: 多轮决策循环（思考→行动→观察→思考），可以调用工具，自动完成复杂任务

### Q4: 如何处理 LLM 应用的高延迟？
- 流式输出（用户快速看到部分结果）
- 缓存（相同/相似查询命中缓存）
- 预计算（提前生成热门内容）
- 异步处理（后台生成，通知用户）
- 选用更快的模型（Haiku 级别）

### Q5: 向量数据库选型考虑因素？
- 数据规模（百万 vs 十亿级向量）
- 查询延迟要求
- 是否需要混合搜索（向量 + 关键词）
- 部署方式（托管 vs 自部署）
- 成本
- 是否需要持久化和实时更新
