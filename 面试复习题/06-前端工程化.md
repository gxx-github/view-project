## 一、构建工具

### 1. Webpack 核心概念
```javascript
// webpack.config.js
module.exports = {
  entry: './src/index.js',       // 入口
  output: {                      // 输出
    path: path.resolve('dist'),
    filename: '[name].[contenthash].js',
    clean: true,
  },
  module: {
    rules: [                     // Loader：处理非 JS 文件
      { test: /\.css$/, use: ['style-loader', 'css-loader', 'postcss-loader'] },
      { test: /\.tsx?$/, use: 'ts-loader', exclude: /node_modules/ },
      { test: /\.(png|jpg)$/, type: 'asset/resource' },
    ],
  },
  plugins: [                     // Plugin：扩展构建能力
    new HtmlWebpackPlugin({ template: './public/index.html' }),
    new MiniCssExtractPlugin(),
    new DefinePlugin({ 'process.env.NODE_ENV': JSON.stringify('production') }),
  ],
  resolve: {
    extensions: ['.tsx', '.ts', '.js'],
    alias: { '@': path.resolve('src') },
  },
  optimization: {
    splitChunks: {               // 代码分割
      chunks: 'all',
      cacheGroups: {
        vendor: { test: /node_modules/, name: 'vendor' },
      },
    },
  },
  devServer: {                   // 开发服务器
    hot: true,
    port: 3000,
    proxy: { '/api': 'http://localhost:8080' },
  },
};
```

### 2. Loader vs Plugin
| Loader | Plugin |
|--------|--------|
| 文件转换器 | 构建流程钩子 |
| 在 module.rules 配置 | 在 plugins 数组配置 |
| 链式调用，从右到左 | 基于 Tapable 事件系统 |
| 如: babel-loader, css-loader | 如: HtmlWebpackPlugin, DefinePlugin |

### 3. Webpack 构建流程
```
1. 初始化参数 — 合并配置文件 + 命令行参数
2. 开始编译 — 创建 Compiler 对象，注册插件
3. 确定入口 — 从 entry 开始
4. 编译模块 — 调用 Loader 转换文件，使用 acorn 解析 AST
5. 依赖解析 — 递归解析 import/require，构建依赖图
6. 生成 chunk — 根据依赖图组装 chunk
7. 输出文件 — 根据 output 配置写入文件系统
```

### 4. Vite 为什么快？
```
开发环境:
  Webpack: 先打包所有模块 → 启动服务器（项目越大越慢）
  Vite:    启动服务器 → 按需编译（利用浏览器 ESM）

  Vite 快的原因:
  1. 原生 ESM — 浏览器直接 import，无需打包
  2. esbuild 预构建 — node_modules 用 esbuild（Go 编写）极速处理
  3. 按需编译 — 只编译当前页面需要的模块
  4. HMR 精确 — 基于 ESM，修改某模块只需替换该模块

生产环境:
  Vite 使用 Rollup 打包（Tree-shaking 更好）
```

### 5. Vite 配置
```typescript
// vite.config.ts
import { defineConfig } from 'vite';
import react from '@vitejs/plugin-react';

export default defineConfig({
  plugins: [react()],
  resolve: {

    rollupOptions: {
      output: {

        },
      },
    },
  },

```

### 6. 构建工具对比
| 特性 | Webpack | Vite | Rollup | esbuild |
|------|---------|------|--------|---------|
| 定位 | 应用打包器 | 开发服务器+打包 | 库打包器 | 极速编译器 |
| 开发体验 | 较慢 | 极快 | - | - |
| Tree-shaking | 支持 | 支持（Rollup） | 最佳 | 支持 |
| 代码分割 | 强 | 支持 | 支持 | 支持 |
| 生态 | 最丰富 | 快速增长 | 库开发丰富 | 插件较少 |
| 适用场景 | 大型应用 | 现代项目 | 库/组件库 | 预处理/构建加速 |

### 2. 错误监控
```javascript
// JS 运行时错误
window.onerror = (msg, url, line, col, error) => {
  reportError({ type: 'js_error', msg, url, line, col, stack: error?.stack });
};

// Promise 未捕获错误
window.addEventListener('unhandledrejection', (event) => {
  reportError({ type: 'promise_error', reason: event.reason });
});

// React 错误边界
class ErrorBoundary extends React.Component {
  componentDidCatch(error, info) {
    reportError({ type: 'react_error', error, componentStack: info.componentStack });
  }
}

// 资源加载错误
window.addEventListener('error', (event) => {
  if (event.target?.src || event.target?.href) {
    reportError({ type: 'resource_error', url: event.target.src || event.target.href });
  }
}, true); // 捕获阶段
```

## 八、工程化面试高频题

### Q1: Webpack HMR 原理？
1. Webpack dev server 与浏览器建立 WebSocket 连接
2. 文件修改触发重新编译，生成 hot-update.json 和 hot-update.js
3. 通过 WebSocket 通知浏览器有更新
4. 浏览器下载增量文件
5. HMR Runtime 替换旧模块，执行 `module.hot.accept` 回调

### Q2: Tree-shaking 原理？
- 基于 ES Module 的静态分析（`import/export` 是静态声明）
- 构建时标记未使用的导出
- 依赖 Terser/SWC 在压缩阶段移除 dead code
- 副作用标记：`package.json` 中 `"sideEffects": false`

### Q3: Source Map 原理？
- 打包时生成映射文件（`.map`），记录编译后代码到源码的映射关系
- 开发环境：inline source map（嵌入 bundle）
- 生产环境：hidden source map（不上传到 CDN，仅用于错误平台）

### Q4: 前端模块化演进？
```
全局变量 → IIFE → CommonJS（Node）→ AMD（RequireJS）→ UMD → ES Module
```
ES Module 是最终标准：静态分析、Tree-shaking、浏览器原生支持。
