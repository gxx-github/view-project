# 微任务与宏任务：执行机制 + 例题精讲

> 面试高频：事件循环、微任务/宏任务顺序。本文讲清机制并附多道例题与解析。

---

## 一、执行机制速记

### 1. 两种任务

| 类型   | 常见来源                     |
|--------|------------------------------|
| **宏任务** | `setTimeout`、`setInterval`、I/O、UI 渲染、`requestAnimationFrame`（部分环境） |
| **微任务** | `Promise.then/catch/finally`、`queueMicrotask`、`MutationObserver`、`async/await`（await 后的代码相当于 .then） |

### 2. 事件循环怎么跑（一句话）

- **每一轮**：先执行完当前**同步代码** → 再**清空当前所有微任务** → 再取**一个宏任务**执行（若该宏任务里又注册了微任务，会在这一个宏任务执行完后立刻把这些微任务跑完，再取下一个宏任务）。
- 更精确的说法：**执行一个宏任务 → 执行期间产生的微任务全部执行完 → 再执行下一个宏任务**。所以「微任务总在下一轮宏任务之前被清空」。

### 3. 两个要点

1. **同步代码最先执行**（包括 `new Promise(executor)` 里的 `executor` 函数，它是同步调用的）。
2. **微任务优先于宏任务**：同一轮里，所有已入队的微任务会在下一个宏任务之前全部执行完。

## 二、你给的例题（先热手）

```javascript
console.log('输出1')
setTimeout(() => {
  console.log("输出2")
})
console.log('输出3')
const promise1 = new Promise((resolve) => {
  console.log('输出4')
  resolve(1)
})
promise1.then(() => {
  console.log("输出5")
  setTimeout(() => {
    console.log("输出6")
  })
})
```

**执行顺序：1 → 3 → 4 → 5 → 2 → 6**

**逐步解析：**

| 步骤 | 动作 | 输出/结果 |
|------|------|-----------|
| 1 | 同步：`console.log('输出1')` | **输出1** |
| 2 | 同步：`setTimeout(...)` 只是把回调**登记**为宏任务，不执行 | 无 |
| 3 | 同步：`console.log('输出3')` | **输出3** |
| 4 | 同步：执行 `new Promise` 的 **executor**，即 `(resolve)=>{ console.log('输出4'); resolve(1) }` | **输出4** |
| 5 | `resolve(1)` 后，`promise1.then` 的回调被放入**微任务队列** | 无 |
| 6 | 当前同步代码执行完毕，开始清空微任务队列 | 无 |
| 7 | 执行微任务：`then` 回调 → `console.log("输出5")`，再 `setTimeout(..., 输出6)` 登记宏任务 | **输出5** |
| 8 | 微任务清空，取下一个宏任务：第一个 `setTimeout`（输出2） | **输出2** |
| 9 | 再取下一个宏任务：输出6 的 `setTimeout` | **输出6** |

**记忆点：**  
- `new Promise(executor)` 里的代码是**同步**的。  
- `then` 回调是**微任务**。  
- `setTimeout` 回调是**宏任务**，所以 2 和 6 一定在 5 之后。

## 三、例题 2：微任务 vs 宏任务（基础）

```javascript
console.log('1')
setTimeout(() => console.log('2'), 0)
Promise.resolve().then(() => console.log('3'))
console.log('4')
```

**顺序：1 → 4 → 3 → 2**

- 同步：1、4。
- 微任务：3（`Promise.then`）。
- 宏任务：2（`setTimeout`）。  
微任务先于宏任务，所以 3 在 2 前。

## 四、例题 3：多层 then + setTimeout

```javascript
console.log('A')
setTimeout(() => console.log('B'), 0)
Promise.resolve()
  .then(() => {
    console.log('C')
    setTimeout(() => console.log('D'), 0)
  })
  .then(() => console.log('E'))
console.log('F')
```

**顺序：A → F → C → E → B → D**

| 阶段 | 动作 | 输出 |
|------|------|------|
| 同步 | A, F | A, F |
| 微任务 | 第一个 then：输出 C，并登记 setTimeout(D) | C |
| 微任务 | 第二个 then（前一个 then 的返回值是 undefined，相当于 resolve(undefined)，所以下一个 then 会进微任务） | E |
| 宏任务 | 第一个 setTimeout | B |
| 宏任务 | 第二个 setTimeout（在 then 里登记的） | D |

要点：每个 `then` 都会在「当前微任务执行完」后可能再往微任务队列里加任务；等所有微任务都跑完，才执行宏任务 B、D。

## 五、例题 4：async/await（本质是 Promise + 微任务）

```javascript
async function fn() {
  console.log('A')
  await console.log('B')  // await 后面是同步的，但 await 后面的代码相当于放在 then 里
  console.log('C')
}
console.log('D')
fn()
console.log('E')
```

**顺序：D → A → B → E → C**

- `await console.log('B')`：`await` 后面是同步表达式，先执行，输出 B；**await 之后的代码**（`console.log('C')`）会被包装成「微任务」。
- 所以：同步 D → 调用 fn() → 同步 A → 同步 B → 同步 E → 微任务 C。

等价理解：

```javascript
// 可理解为：
console.log('D')
console.log('A')
console.log('B')
Promise.resolve().then(() => console.log('C'))  // await 后的代码
console.log('E')
```

## 六、例题 5：async + Promise 混排

```javascript
console.log('1')
async function async1() {
  console.log('2')
  await async2()
  console.log('3')
}
async function async2() {
  console.log('4')
}
async1()
new Promise((resolve) => {
  console.log('5')
  resolve()
}).then(() => {
  console.log('6')
})
console.log('7')
```

1 2 4 5 7 3 6


**顺序：1 → 2 → 4 → 5 → 7 → 3 → 6**

| 步骤 | 说明 |
|------|------|
| 同步 | 1 |
| 调用 async1()，进入 async1：同步 2；遇到 await async2()，先执行 async2()，同步 4；await 后的 3 被放入微任务 |
| 同步 | new Promise 的 executor 执行，输出 5，resolve() 使 then 回调入微任务队列 |
| 同步 | 7 |
| 微任务 | 先执行 await 后面的 3，再执行 then 的 6（谁先入队先执行，这里一般是 3 先 6 后） |

## 七、例题 6：queueMicrotask 明确是微任务

```javascript
console.log('a')
setTimeout(() => console.log('b'), 0)
queueMicrotask(() => console.log('c'))
Promise.resolve().then(() => console.log('d'))
console.log('e')
```

**顺序：a → e → c → d → b**

- 同步：a、e。
- 微任务按入队顺序：c、d。
- 宏任务：b。

## 八、例题 7：resolve 里再包 Promise

```javascript
console.log('start')
Promise.resolve()
  .then(() => {
    console.log('p1')
    return Promise.resolve('p2')  // 多了一层 Promise，会多一次微任务
  })
  .then((r) => console.log(r))   // 这里的 r 是 'p2'，输出就是顺序里的 p2
Promise.resolve().then(() => console.log('p3'))
console.log('end')
```

**顺序：start → end → p1 → p3 → p2**（其中 p2 来自第二个 then 的 `console.log(r)`）

- 同步：start、end。
- 第一个 then 输出 p1，并 `return Promise.resolve('p2')`。在规范里，**then 的返回值如果是 Promise，会多插入一次微任务**来解析这个 Promise，所以第二个 then 会晚于「当前已入队的 p3」执行。
- 微任务顺序：p1 → p3 →（解析 return 的 Promise 产生的微任务）→ **第二个 then 的 `console.log(r)` 执行，输出 p2**。

## 九、例题 8：综合大杂烩

```javascript
console.log('0')
setTimeout(() => {
  console.log('1')
  Promise.resolve().then(() => console.log('2'))
}, 0)
setTimeout(() => console.log('3'), 0)
Promise.resolve().then(() => {
  console.log('4')
  setTimeout(() => console.log('5'), 0)
})
console.log('6')
```

**顺序：0 → 6 → 4 → 1 → 2 → 3 → 5**

| 阶段 | 动作 |
|------|------|
| 同步 | 0, 6；两个 setTimeout(1、3) 和 then 回调只登记，不执行 |
| 微任务 | then 执行：4，并登记 setTimeout(5) |
| 宏任务 1 | 第一个 setTimeout：输出 1，内部 Promise.then 入微任务；微任务立刻被清空，输出 2 |
| 宏任务 2 | 第二个 setTimeout：3 |
| 宏任务 3 | 在 then 里登记的 setTimeout：5 |

**规则再强调：一个宏任务执行完后，如果产生了新的微任务，会先把这些微任务全部执行完，再执行下一个宏任务。**

## 十、小结：答题口诀

1. **同步永远最先**（包括 `new Promise(executor)` 和 `await` 右侧的表达式）。
2. **微任务在「当前宏任务之后、下一个宏任务之前」全部清空**。
3. **同一类型**：微任务按入队顺序、宏任务按入队顺序。
4. **async/await**：await 后面的代码 ≈ 放在 `Promise.then` 里，即微任务。
5. **then 里 return Promise**：会多一次微任务再执行后面的 then。
