## 一、先回答核心问题：这个项目用到了 Agent 吗？

**严格来说，没有。**

但是用到了 Agent 的一项核心能力 — **Tool Calling（工具调用）**。

打个比方来解释三者的区别：

### 普通聊天（没有工具）
> 你问 ChatGPT："推荐几部甜剧"
> 它直接回答一段文字
> 就像你问朋友"吃什么"，朋友凭经验回答"去吃火锅吧"

### Tool Calling 工具调用（本项目 ✅）
> 你问："推荐几部甜剧"
> AI 不直接打字回答，而是**调用一个叫 renderDramaCard 的工具**
> 把推荐内容填进一张结构化的"表格"里
> 前端拿到这张"表格"，渲染成精美的卡片
> 就像你问朋友"吃什么"，朋友**填了一张外卖单**给你，上面有菜名、价格、评分

### Agent 智能体（更高级 ❌ 本项目没用）
> 你问："帮我规划周末追剧计划"
> Agent 会**自主决定**：
> 1. 先调用"搜索工具"查最新热播剧
> 2. 再调用"评分工具"查豆瓣分数
> 3. 再调用"日历工具"看你周末有没有空
> 4. 最后综合所有信息给你一个完整方案
> 就像你问朋友"周末怎么安排"，朋友**自己去查了天气、查了餐厅、看了电影场次**，然后给你一个完整计划

**一句话总结：**
| 模式 | 本项目用了？ | 类比 |
|------|-------------|------|
| 普通聊天 | 基础能力 | 朋友口头推荐 |
| Tool Calling | ✅ 用了 | 朋友填了一张表格给你 |
| Agent | ❌ 没用 | 朋友自己去调研后给你完整方案 |

## 二、整个项目是怎么工作的？（用外卖打个比方）

想象你在用**外卖 APP 点奶茶**：

```
你（用户）
  │
  │ 在输入框打字："想看点甜甜的剧"
  │
  ▼
┌─────────────────────────────────┐
│  前端页面（手机界面）             │
│                                 │
│  输入框就像外卖APP的搜索栏        │
│  消息列表就像订单记录             │
│                                 │
│  useChat 是一个"自动接单员"      │
│  它负责：                        │
│  · 把你的话发给后厨（后端）       │
│  · 等后厨做好，端上来给你看        │
│  · 如果后厨传回来的是"卡片"       │
│    就帮你摆盘成漂亮的样式         │
└──────────────┬──────────────────┘
               │
               │ 相当于：下单！
               │
               ▼
┌─────────────────────────────────┐
│  后端 API（后厨）                │
│                                 │
│  route.ts 就是后厨的厨师长       │
│  它负责：                        │
│  1. 接到你的单（消息）            │
│  2. 把菜单翻译成厨师能看懂的格式   │
│  3. 告诉 AI 大厨："客人要甜剧！"   │
│                                 │
│  streamText 就是"下单给AI大厨"   │
│  tools 就是给大厨准备的"模具"     │
│  - renderDramaCard 这个模具      │
│    告诉大厨：把推荐内容填进        │
│    这个固定格式的表格里            │
│                                 │
│  stepCountIs(3) 就像"最多做3轮"  │
│  · 第1轮：AI 看你要什么           │
│  · 第2轮：AI 调用工具填表格       │
│  · 第3轮：AI 看看还要不要补充     │
└──────────────┬──────────────────┘
               │
               │ 相当于：后厨把单子交给AI大厨
               │
               ▼
┌─────────────────────────────────┐
│  智谱 AI（AI 大厨）              │
│                                 │
│  GLM-4-Plus 就是一个"超级大厨"   │
│  它读了无数剧本，记住了所有电视剧   │
│                                 │
│  system prompt 就像"餐厅的定位"  │
│  告诉大厨："你是专门推荐甜剧的"   │
│                                 │
│  大厨不是直接写一段文字，而是：     │
│  1. 理解你的心情（"工作太累了"）   │
│  2. 想到几部合适的甜剧             │
│  3. 调用 renderDramaCard 工具     │
│     把信息填进表格：               │
│     {                            │
│       mood: "辛苦了～来点甜的",   │
│       dramas: [                  │
│         {                        │
│           title: "一闪一闪亮星星",│
│           year: "2022",          │
│           sweetness: 5,          │
│           tags: ["高甜","青春"],  │
│           reason: "超甜超心动"    │
│         }                        │
│       ]                          │
│     }                            │
│                                 │
│  这个过程是"边做边上菜"的         │
│  也就是 SSE 流式传输              │
│  大厨切好一片就端上来一片          │
│  你不需要等整桌菜做完              │
└─────────────────────────────────┘
```

## 三、关键技术概念，一个一个讲

### 1. "伪装成 OpenAI" 是什么意思？

想象所有的 AI 模型都说不同的"方言"：
- OpenAI 说"OpenAI 方言"
- 智谱说"智谱方言"
- Claude 说"Claude 方言"

但 Vercel AI SDK 只听得懂"OpenAI 方言"。

幸运的是，**智谱 GLM-4 的 API 接口和 OpenAI 一模一样**，就像两个地方的人说同样的方言。

所以我们只需要改一个地址：

```typescript
// 原来指向 OpenAI 的服务器
baseURL: "https://api.openai.com/v1"

// 改成指向智谱的服务器（接口格式一模一样！）
baseURL: "https://open.bigmodel.cn/api/paas/v4"
```

**价值：** 以后想换模型，改一行地址就行，代码不用动。

---

### 2. Tool Calling 到底做了什么？

**没有工具时：**
```
AI 返回 → "我推荐你看一闪一闪亮星星，特别甜..."
前端显示 → 一段文字（跟普通聊天没区别）
```

**有工具时：**
```
AI 返回 → {
  tool: "renderDramaCard",
  data: {
    dramas: [
      { title: "一闪一闪亮星星", sweetness: 5, tags: ["高甜"] }
    ]
  }
}
前端拿到结构化数据 → 渲染成精美卡片（有甜度条、有彩色标签、有推荐理由）
```

**这就是 Generative UI 的核心思想：AI 决定用什么 UI 组件来展示信息，而不是永远只返回一段文字。**

---

### 3. zod / zodSchema 是干什么的？

zod 是一个**数据校验器**，就像餐厅里的"菜品质量检查员"。

我们用 zod 告诉 AI："你的推荐必须包含这些字段，类型必须是这样"：

```
剧名(title) → 必须是文字
年份(year) → 必须是文字
甜度(sweetness) → 必须是数字 1-5
标签(tags) → 必须是文字列表
推荐理由(reason) → 必须是文字
```

这样 AI 就不会返回乱七八糟的东西，前端拿到数据后可以放心渲染。

---

### 4. SSE 流式传输是什么？

**普通请求：**
```
你提问 → 等待 10 秒 → 一次性收到完整回答
（就像下载一个文件，下完才能打开）
```

**SSE 流式：**
```
你提问 → 0.5 秒后收到第一个字 → 0.6 秒收到第二个字 → ...
（就像看在线视频，边下边播）
```

我们用的 `streamText` + `toUIMessageStreamResponse()` 就是告诉服务器：
"别等做完再端上来，边做边给我，我实时显示给用户"

---

### 5. useChat Hook 做了什么？

`useChat` 是 Vercel AI SDK 提供的一个 React Hook，它就像一个**全自动的客服系统**：

| 你要做的 | useChat 自动做的 |
|---------|----------------|
| 用户输入文字 | sendMessage() 发给后端 |
| 等待回复 | 自动接收 SSE 流 |
| 显示回复 | 自动更新 messages 数组 |
| 显示加载状态 | 自动管理 status 状态 |
| 工具调用 | 自动解析 parts 里的 tool 数据 |

你只需要写"拿到数据后怎么展示"的代码，所有通信逻辑它全包了。

---

### 6. 什么是 ai@6 的 breaking changes？

Vercel AI SDK 升级到第 6 版时，改了很多 API，就像微信大更新改了界面一样：

**旧版（ai@3）：**
```javascript
// 消息结构
{ role: "assistant", content: "推荐你看..." }

// 发送消息
handleSubmit(event)
```

**新版（ai@6）：**
```javascript
// 消息结构（content 没了，变成 parts 数组）
{ role: "assistant", parts: [{ type: "text", text: "推荐你看..." }] }

// 发送消息（没有 handleSubmit 了）
sendMessage({ text: "..." })
```

本项目完整适配了 ai@6 的所有变化，这是一个实际的工程能力体现。

## 四、项目的数据流（完整版）

```
用户打字 "想看点甜甜的"
         │
         ▼
① sendMessage({ text: "想看点甜甜的" })
   useChat 把消息加入 messages 数组
   同时发起 POST /api/chat 请求
         │
         ▼
② route.ts 收到请求
   convertToModelMessages() 把前端格式转为模型格式
   streamText() 调用智谱 GLM-4-Plus
         │
         ▼
③ 智谱 AI 处理
   AI 看到你有 renderDramaCard 工具
   AI 决定调用这个工具（而不是纯文本回复）
   返回工具调用数据（包含剧名、甜度等）
         │
         ▼
④ execute() 执行工具
   把 AI 填好的数据原样返回
   （我们的工具只是"格式化数据"，不调外部API）
         │
         ▼
⑤ AI 拿到工具结果后，可能再生成一段文字
   （这就是 stepCountIs(3) 允许多步的原因）
   最终生成完整的响应
         │
         ▼
⑥ toUIMessageStreamResponse() 以 SSE 流返回
   包含：工具调用数据 + 可能的后续文字
         │
         ▼
⑦ useChat 自动解析 SSE 流
   messages 数组中新增 assistant 消息
   消息的 parts 包含：
   · { type: "tool-renderDramaCard", state: "output-available", output: {...} }
   · 可能还有 { type: "text", text: "..." }
         │
         ▼
⑧ 前端渲染
   遍历 msg.parts：
   · 遇到 text → 渲染文字气泡
   · 遇到 tool-renderDramaCard → 渲染 DramaCard 组件
     · 剧名、年份、甜度🍬、彩色标签、推荐理由
```

## 五、这个项目的价值（面试怎么说）

### 不是简单的 ChatGPT 套壳
- 用了 Tool Calling 实现结构化输出
- 前端根据 AI 返回的数据类型动态决定渲染组件
- 这就是 **Generative UI** 的核心思想

### 适配了最新的 ai@6
- 很多人还在用 ai@3 的旧 API
- 本项目完整适配了 ai@6 的 breaking changes
- 体现了阅读源码和跟进新技术的能力

### Provider 适配器模式
- 一行代码切换国内外模型
- 理解了 AI SDK 的抽象层设计
- 不是简单的 API 调用，而是架构层面的理解

### 全栈能力
- 前端：React + Tailwind + 毛玻璃 UI
- 后端：Next.js API Route + SSE 流式
- AI：Tool Calling + Schema 校验 + System Prompt 设计
- 一人搞定端到端
