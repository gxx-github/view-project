# AI 影视推荐助手（AIDEMO）— 简历项目描述

---

## 一、项目概述

基于 Next.js 14 + Bun 的 AI 影视推荐助手，用户通过自然语言对话获取个性化影视剧推荐。集成 RAG 语义检索 + 多数据源 API + AI Agent 工具调用，实现智能搜索、对比、心情推荐等功能。

**定位**: 前端为主的全栈 AI 应用，涵盖前端开发 + API 路由 + 数据库 + AI 集成

## 四、核心模块详解

### 4.1 AI Agent 工具调度系统

**文件**: `src/lib/agent.ts`、`src/lib/tools/`

**技术栈**: Vercel AI SDK、智谱 GLM-4-Plus、Tool Calling

**功能**:

- AI Agent "甜甜" 根据用户自然语言意图，自动选择并调用对应工具
- 10 个工具覆盖搜索、对比、推荐、详情等场景
- 工具执行结果返回 Agent 继续推理，形成多轮工具调用链

**工具清单**:


| 工具                                                             | 数据源        | 功能             |
| -------------------------------------------------------------- | ---------- | -------------- |
| searchDramas                                                   | TVmaze API | 按关键词精确搜索剧集     |
| ragSearch                                                      | 本地向量库      | 语义搜索（"甜甜的恋爱剧"） |
| compareDramas                                                  | 本地数据       | 两部剧多维度对比       |
| analyzeMood                                                    | LLM 推理     | 根据用户心情推荐剧集     |
| getTrending                                                    | 本地数据       | 获取热门推荐列表       |
| tmdbSearch / tmdbGetDetails / tmdbGetTrending / tmdbGetByActor | TMDB API   | TMDB 数据源查询     |
| renderDramaCard                                                | -          | 结构化渲染剧集卡片      |


**亮点**:

- **双路搜索策略**: 精确搜索走 TVmaze/TMDB API，模糊语义搜索走本地 RAG 向量检索，Agent 自动判断用哪条路
- **工具链组合**: 用户一句话可能触发多个工具串联执行（如 "帮我找一部类似《去有风的地方》的剧" → ragSearch → tmdbGetDetails）

---

### 4.2 RAG 语义检索系统

**文件**: `src/lib/rag/`

**技术栈**: 智谱 embedding-3 向量模型、余弦相似度、Top-K 检索

**功能**:

- 将 27 部影视剧的元数据（名称、类型、简介、评分、演员）向量化存储
- 用户输入自然语言描述后，计算与所有剧集的语义相似度，返回 Top-K 结果
- 按需加载：首次请求时才构建向量索引

**流程**:

```
27部剧元数据 → 智谱 embedding-3 向量化 → 存储到 vector-store
                                                  ↓
用户输入 "适合周末看的轻松甜宠剧" → 向量化 → 余弦相似度计算 → Top-5 结果
```

**亮点**:

- 纯前端项目也能落地 RAG，无需部署向量数据库（SQLite + 内存向量）
- 语义理解能力强：用户说"甜甜的恋爱剧"能匹配到《去有风的地方》，精确搜索做不到

---

### 4.3 多会话持久化

**文件**: `src/hooks/usePersistedChat.ts`、`src/hooks/useSessionManager.ts`、`src/lib/storage.ts`

**技术栈**: React Hook、localStorage

**功能**:

- 多会话管理：新建、切换、删除、重命名对话
- 会话搜索：按标题关键词搜索历史会话
- 消息持久化：所有聊天记录自动保存到 localStorage
- 页面刷新不丢失数据

**亮点**:

- 自定义 Hook 封装，业务逻辑与 UI 完全分离
- 会话切换时消息无缝恢复，用户体验接近 ChatGPT

## 八、面试高频问题 & 参考回答

### Q: "RAG 是怎么实现的？"

> "三部曲：Indexer 把数据向量化存起来，Retriever 负责检索。数据是 27 部剧的元数据，调用智谱的 embedding-3 模型转成向量存在内存里。用户 query 也做一次 embedding，然后和所有剧集向量算余弦相似度，取 Top-K。向量维度是智谱模型决定的，存储用一个简单的数组，检索时逐个算相似度。"

### Q: "为什么不用 Pinecone 这种向量数据库？"

> "数据量小，27 部剧的元数据向量完全可以放内存。用 Pinecone 还要额外部署、付费，对于这种规模的数据内存检索就够了，检索速度也很快。如果数据量到万级以上再考虑用专用向量数据库。"

### Q: "Agent 的工具是怎么调度的？"

> "用的 Vercel AI SDK 的 Tool Calling 能力。每个工具用 TypeScript 定义 name、description、parameters（JSON Schema 格式）。用户发消息后，LLM 根据对话上下文决定调哪个工具、传什么参数，SDK 自动执行工具函数，结果返回给 LLM 继续推理。如果需要，LLM 可以连续调用多个工具。"

### Q: "双路搜索怎么判断走哪条路？"

> "不用我手动判断，交给 Agent 自己决策。我在工具描述里写清楚了每个工具的特点——searchDramas 适合精确搜索已知剧名，ragSearch 适合模糊语义描述。LLM 根据用户输入自行判断调哪个。实际测试下来准确率很高。"

### Q: "Drizzle ORM 和 Prisma 有什么区别？"

> "Drizzle 更轻量，API 更贴近 SQL，生成的查询体积更小。Prisma 有自己的 Schema 文件和查询引擎，功能更全但更重。对于这个项目，Drizzle 的类型推断和零运行时开销更适合，而且和 Bun 的兼容性更好。"

### Q: "Bun 和 Node.js 的区别？"

> "Bun 是用 Zig 语言写的 JavaScript 运行时，兼容 Node.js API 但速度更快。最大的感受是 `bun install` 比 `npm install` 快很多，冷启动也更快。API 完全兼容，迁移成本很低，基本上把 node 换成 bun 就行。"

### Q: "流式响应是怎么实现的？"

> "用的 Vercel AI SDK 的 useChat Hook。后端 API Route 里调用 AI SDK 的 streamText 方法，SDK 会把 LLM 的流式输出逐 token 转成 Server-Sent Events 推给前端。前端 useChat 自动消费 SSE 流，更新响应式状态，UI 实时渲染。不需要自己处理 WebSocket 或手动解析流。"

### Q: "Prompt 注入检测怎么做的？"

> "在后端 API Route 里做了简单的规则检测。比如检查用户输入里是否包含'忽略之前的指令'、'你现在是'这类常见的注入模式。匹配到就拦截请求返回错误。这是一个基础防护，生产环境还需要更多层的检测。"
